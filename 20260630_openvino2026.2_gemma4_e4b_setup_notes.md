## Goal
Convert and benchmark `google/gemma-4-E4B-it-qat-q4_0-unquantized` to OpenVINO INT4 IR
using the **official** OV 2026.2 release (no custom PR build).

## Environment
- **OpenVINO GenAI**: 2026.2.0.0 (from `C:\working\gemma4\openvino_genai_windows_2026.2.0.0_x86_64`)

## Conversion Command (Working)
```powershell
& c:\working\gemma4\.venv\Scripts\Activate.ps1
cd C:\working\gemma4\gemma4-openvino-genai\models
optimum-cli export openvino `
    --model google/gemma-4-E4B-it-qat-q4_0-unquantized `
    --task image-text-to-text `
    --weight-format int4 `
    --group-size 64 `
    --ratio 1.0 `
    gemma-4-E4B-it-ov-qat
```

**Key flags:**
- `--task image-text-to-text` — **required**, gemma4 auto-detects as `any-to-any` which is rejected
- `--weight-format int4 --group-size 64 --ratio 1.0` — NNCF INT4 compression (72% int4_asym / 28% int8_asym)

## Issues Encountered & Fixes Applied

### Fix 1 — `transformers` upgraded 5.0.0 → 5.12.1
**Error:** `KeyError: 'gemma4'` in `AutoConfig.from_pretrained`  
**Root cause:** `gemma4` model type was not registered in `CONFIG_MAPPING` in transformers 5.0.0.
Gemma4 support was added in transformers ~4.51+.  
**Fix:**
```powershell
pip install "transformers==5.12.1" --force-reinstall --no-deps
```

### Fix 2 — `optimum-intel` upgraded dev commit → 2.0.0 official
**Error:** `ImportError: cannot import name 'MambaCache' from transformers.models.mamba.modeling_mamba`  
**Root cause:** The pinned dev commit in `export-requirements.txt` (`f50a0b3`) was built against
transformers 5.0.0 which still had `MambaCache`. It was removed in 5.x.  
**Fix:**
```powershell
pip install "optimum-intel[openvino]==2.0.0"
pip install "transformers==5.12.1" --force-reinstall --no-deps  # re-pin after optimum-intel drags it back
```

### Fix 3 — `modeling_decoder.py` MambaCache import stub
**File:** `.venv/Lib/site-packages/optimum/intel/openvino/modeling_decoder.py` (line ~34)  
**Error:** `optimum-intel 2.0.0` still imports `MambaCache` at module level  
**Fix:** Added try/except stub around the import:
```python
try:
    from transformers.models.mamba.modeling_mamba import MambaCache
except ImportError:
    MambaCache = None  # removed in transformers 5.x
```

### Fix 4 — `model_patcher.py` `kv_shared_layer_index` AttributeError
**File:** `.venv/Lib/site-packages/optimum/exporters/openvino/model_patcher.py`  
**Error:** `AttributeError: 'Gemma4TextAttention' object has no attribute 'kv_shared_layer_index'`  
**Root cause:** `optimum-intel 2.0.0`'s Gemma4 patcher stored/retrieved shared KV cache using
`self.kv_shared_layer_index` (a numeric index). In transformers 5.12.1,
`Gemma4TextAttention` no longer sets this attribute — the new API uses `self.layer_type`
(string, e.g. `"global_attention"`) as the dict key for `past_key_values.shared_layers`.  
**Fix:** Patched `Gemma4LMModelPatcher.__enter__` to use `layer_type` as the key for both
storing and retrieving shared KV states, matching the new transformers API.

### Fix 5 — `--task image-text-to-text` must be explicit
**Error:** `ValueError: Asked to export a gemma4 model for the task any-to-any (auto-detected), but the Optimum OpenVINO exporter only supports the tasks image-text-to-text for gemma4`  
**Root cause:** Auto task detection on the QAT model (`q4_0-unquantized`) resolves to `any-to-any`
because its config has additional fields that confuse the task inferrer.  
**Fix:** Always pass `--task image-text-to-text` explicitly.

### Why the sizes are identical despite "QAT"

The model name `gemma-4-E4B-it-qat-q4_0-**unquantized**` is key:
- `qat` = trained with Quantization-Aware Training (the model learned to be robust to INT4)
- `q4_0` = target quantization format
- `unquantized` = the weights are still stored in **BF16** (original precision) — not yet quantized

When we run `optimum-cli export openvino --weight-format int4`, NNCF applies the
**same** INT4 post-training compression to both models. The OV INT4 IR output is therefore
identical in size — the QAT training only improves **accuracy** at INT4, not the file size.

**To get an actually smaller file**, you would use the pre-quantized variant:
`google/gemma-4-E4B-it-qat-w4a16-ct` — this has weights already stored as INT4 (w4a16 =
weight-4bit, activation-16bit, ct = compressed tensor). Exporting that with
`--weight-format int4` passes the pre-quantized weights through directly without NNCF
re-compression, and the resulting IR would be the same ~2.6 GB but loaded faster
(no NNCF calibration step).

The accuracy benefit of QAT over standard PTQ (post-training quantization) is typically
**0.5–2 perplexity points** better at INT4 precision.
